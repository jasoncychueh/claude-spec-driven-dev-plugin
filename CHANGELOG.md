## 1.14.0 (2026-07-17)

Make the backlog **safe for concurrent sessions**, on two independent axes. 1.11.0 gave deferred discoveries a durable home but assumed one writer: item ids came from a counter, and a picked-up item carried no claim. Both assumptions break the moment two branches / worktrees / terminals touch the backlog — and they break in different ways, so they get different fixes.

- **Ids are random hashes, not sequential numbers** — `bl-{NNNN}` → `bl-{hash}` (6 hex chars). The real defect was never the *format*: it was that a counter forces every writer to read global state first ("scan root + archive for the highest existing id"), so two sessions that scan concurrently both see `bl-0008` and both write `bl-0009`. **The id generation was the race**; a random id needs no coordination, so there is nothing to race on. That read-the-highest-id scan is deleted outright, replaced by a cheap existence re-check (regenerate on the freak collision).
- **The generator is pinned, and that pin is the whole change** — `[guid]::NewGuid().ToString('N').Substring(0,6)`, stated as mandatory in both `backlog-guide.md` and SKILL.md. A doc that merely said "use a hash" would be actively worse than the counter: the main agent writes items inline, and an LLM asked for six "random" hex characters is not an RNG — it repeats and clusters, so ids would collide exactly as before, except now silently and unfixably. Random GUID-slice over content-hash deliberately: content-hash's only extra gift is dedup of identical items, which nobody needed, at the cost of input assembly and another layer of model judgment.
- **Chronology moves to `date`** — ids no longer imply sequence, and nothing is lost: `date` was already in every index line and every item's frontmatter. `/backlog` never parsed ids numerically, so it needed no change.
- **Picked-up items are claim-marked** — a separate race the ids don't touch: unique ids stop two sessions *recording* the same id, not two sessions *working* the same item. On pick-up, **before the work starts**, the item gets `status: in-progress` + `picked_up: DATE (branch: X) — what's being done`, and the index line flips to `[~]` carrying the same claim inline. Each of the three fields earns its place: date (is this claim minutes old or a month stale?), branch (go look at the work), one sentence (does it overlap with what I was about to do?). Mark-first-work-second — a claim written after the work starts protects nothing during the window where the collision happens.
- **A `[~]` item is never silently taken over or silently skipped** — the finder reports the claim to the user and lets them decide (pick something else / take over an abandoned session / coordinate). **No expiry rule, on purpose**: "older than N days = stale, take it freely" has no defensible N, and a two-week claim can be an active long-running branch while a two-day one is dead. Whether a claim is live is knowledge that exists outside the repo, so it's a genuine user call with no safe default — the one place in this flow where asking beats guessing. `/backlog list`'s existing stale-item flagging already covers the cleanup need without guessing.

## 1.13.0 (2026-07-12)

Extend the advisor gate with a **second application — co-arbitrating reviewer output**. The 1.12.0 gate fired before the main agent interrupted the *user*; this adds the advisor as a second pair of eyes before the main agent commits a consequential judgment of a *subagent's returned output*. Motivation: the generator/arbiter split already has the main agent challenge the reviewer every round, but the *outcome* of that challenge — what gets fixed vs dropped — is still the premium main agent's solo call, and that solo call is exactly where a wrong dismissal or a false convergence does the most damage.

- **Two consultation points, consequential-trigger only** — during the challenge exchange, when the main agent is about to **dismiss/downgrade a reviewer-graded Critical/High as a false positive** or **accept the whole issue list wholesale without a substantive challenge**; and **before declaring convergence** (a `0 issues` loop exit). Routine arbitration (confirm a Critical is real → fix it) needs no advisor — it fires only when the call is high-stakes or the main agent is genuinely unsure, keeping the most-premium tier spent on the judgments that carry risk across a many-round loop.
- **A second judgment over the same evidence, not new visibility** — the advisor sees the reviewer's *returned* issue list + the artifact + the whole session (exactly what the main agent sees), never the reviewer's internal process (that never enters the main agent's transcript). That's the point: the risk is the main agent misjudging the evidence in front of it, and a stronger model re-judging the same evidence is what catches it.
- **No defer tier here** — unlike the escalation gate, this isn't a stop the main agent is trying to avoid; it's a judgment being made anyway, so the advisor simply weighs in before it's committed. Graceful degradation is identical: advisor unavailable → solo arbitration, exactly as today. Compatible with the existing "exactly one challenge per round" discipline (the advisor consult is the main agent's own second opinion, separate from the single reviewer challenge).
- **Recorded for later review, in its correct home** — a *dismissal* the advisor concurred with is annotated `(advisor concurred, DATE)` in review-log §4; a concurred *downgrade* (the issue is still real, just lower severity) stays an issue at its new severity with a §1 note, **not** filed as a false positive; an advisor dissent that flips the call keeps the issue with a one-line §1 note; a convergence the advisor confirmed is noted in the Summary (a `0 issues` round has no §1 row). The consult happens *before* the round's single challenge so it shapes it. No new section. Principle #13 renamed "Consult the Advisor Before a Consequential Call"; the mechanism is threaded across SKILL.md, review-protocol.md, advisor-gate-guide.md (new §8), and review-log-guide.md.
- **The advisor is now a main-agent-only tool, enforced structurally** — subagents were observed calling the advisor themselves, which inverts the generator/arbiter economy (the advisor is the most-premium tier) and gains nothing (a subagent's call would forward only its own narrow transcript, not the whole session). Two prongs: this plugin's seven agents each set `disallowedTools: advisor` in frontmatter (they keep every other tool, including environment-specific MCP servers); and because built-in `Explore` / `general-purpose` agents can't be frontmatter-restricted and the Task tool has no per-call tool-restriction parameter, whoever spawns one adds a line to its spawn prompt telling it not to use the advisor (wired into "Model Economy for Exploration" and the reviewer fan-out discipline).

## 1.12.0 (2026-07-12)

Add the **Advisor Gate**: before the main agent stops to ask the user any decision, it consults the **advisor** first (a stronger reviewer model that sees the full transcript). Motivation: on a premium session model the main agent interrupts the user far more often than the work warrants — many stops are "which of these two technically-equivalent paths" calls a competent engineer would just decide, and each one costs the user a context-switch (and, on an expensive tier, the reluctance to decide is itself expensive). The advisor is a cheaper second opinion that absorbs those stops, reserving the user's attention for the choices that are genuinely theirs.

- **Three-tier ladder** — before any stop-and-ask: (1) consult the advisor → a clear answer applies + is recorded + work continues; (2) no usable answer (genuine preference / product / irreversible call, advisor uncertain, **or the advisor tool unavailable**) → pick the most defensible default and keep going, recording the assumption; (3) genuinely blocking with no safe default → stop and ask the user (today's behavior). Interrupting the user is the last resort, not the reflex.
- **Graceful degradation** — the advisor tool isn't guaranteed to exist and a call can error; on absent/disabled/error the gate **collapses to exactly today's behavior** (defer where safe, otherwise ask). The gate only ever *reduces* interruptions; it never makes the user worse off.
- **Triage, not takeover** — the advisor settles decisions with a defensible technical answer (codebase consistency, standard idiom, disambiguation); it passes through genuine preference/product/business-priority calls, irreversible / high-blast-radius choices (schema migration, data loss, external contract, security posture), and anything it flags as uncertain. This **refines Core Principle #5** (the reviewer still never resolves an Architecture Decision — only what the *main agent* does on receipt changes: route through the gate instead of straight to `AskUserQuestion`). New shared Principle #13; Spec/Quick-Fix-specific principles renumbered 14–17.
- **Stays human** — Plan/Spec briefings and the ExitPlanMode approval are never advisor-gated (the advisor is a technical reviewer, not the product owner — it can't sign off on *what gets built*); the convergence fuse still reports to the user. Steering promotion passes through as a user-confirmed guardrail call (Principle #7); Medium/Low keep defer-and-batch; mode up/downgrade stays the main agent's own informed routing call.
- **Recorded + surfaced (the safety valve)** — an advisor-resolved decision lands in `review-log.md` §2 tagged `[advisor-resolved · pending your review]` with `Rationale (advisor, DATE)` and a new §1 Status `advisor-resolved` (Quick Fix uses the plan file's embedded log). Recording is silent (asking each time would reintroduce the interruption the gate removes); every briefing and Summary then carries a **"Decisions I resolved via the advisor — confirm or override"** block, the one consolidated review point where the user can flip any of them (an override is handled like a fresh user Decision). New `references/advisor-gate-guide.md`; the mechanism + terminology are threaded in sync across SKILL.md, review-protocol.md, decision-escalation-guide.md, review-log-guide.md, the review-log template, briefing-guide.md, checklists.md, and spec-workflow.md.

## 1.11.0 (2026-07-11)

Add a **project backlog** (`.spec/backlog/`): a durable parking lot for anything discovered mid-flow that can't be resolved now or needs deeper discussion later. Motivation: deferred discoveries were dying in three graves — a review waiver buried in one spec's review-log, an out-of-scope finding that existed only in an implementer's completion report, a "worth rethinking later" remark that evaporated with the session. The backlog gives all three one durable home the next session can scan in a single read.

- **MEMORY.md-style structure** — `BACKLOG.md` index (one line per item, **open/in-progress only**) + one `bl-{NNNN}-{slug}.md` file per item (frontmatter: id/title/type/status/date/source/feature; body: Problem / Context / Suggested next step, written for a reader two weeks later) + `archive/` for closed items. Progressive disclosure: the index answers "what do we still owe" in one read; item files carry context thick enough to act on without re-asking the user. Per-item files also keep git diffs collision-free.
- **Backlog vs waiver semantics** — waiver (review-log §3) = "accepted, not a debt"; backlog = "a debt we intend to repay." The Medium/Low defer-and-batch round now offers three outcomes (fix now / waive / handle later → backlog); review-log §1 gains a `backlogged` Status citing the item id (status enumerations and the convergence flow updated in sync across review-log-guide.md, review-log-template.md, review-protocol.md, decision-escalation-guide.md, and spec-workflow.md).
- **Silent writes by the main agent** — recording is arbiter bookkeeping (like review-log maintenance): no subagent, no per-item confirmation — recording is cheap and reversible, a cleanup pass prunes noise, and asking every time would kill the habit. New items surface in the end-of-flow Summary. Three hook points: review Medium/Low batch, `spec-implementer` completion reports (new "Out-of-scope findings" report bullet — the implementer reports, never writes the backlog itself), and plain conversation.
- **One uniform close rule** — set `status` + `resolution:` → move to `archive/` → remove the index line. Archive rather than delete: a `dropped` item's resolution is a lightweight ADR ("considered X, decided no, because Y") that stops the same idea from being re-discovered and re-litigated; `done` gets the same treatment purely to keep one rule. Invariant: BACKLOG.md always equals the exact set of unresolved items.
- **New `/backlog` command** (list / pick \<id\> / close \<id\> / drop \<id\>) — picking an item briefs the user from the item file, then routes it through normal mode selection like any incoming task. `/load-spec` output now shows the open-item count. New `references/backlog-guide.md`; new shared Core Principle #12 "Backlog the Unresolved" (Spec/Quick-Fix-specific principles renumbered 13–16); Steering Evolution gains the channel split — principle-to-follow → steering (restrained, confirmed), work-to-do-later → backlog (liberal, silent).

## 1.10.0 (2026-07-05)

Extend the generator/arbiter economy from **writes to reads**: broad codebase exploration delegated to a built-in `Explore` / `general-purpose` agent no longer inherits the session's top-tier model (which may be Fable — the most premium tier), it gets a **deliberately pinned cheaper tier**. Motivation: the split already kept long-form *generation* off the premium model, but *exploration* — sweeping many files to locate something, map a subsystem, or trace callers before a brief — is equally bulk work, and a broad sweep left to inherit Fable/Opus burned tokens out of all proportion to the judgment it needed. A **known target needs no subagent at all** — read it directly via codebase-memory MCP / a targeted `Grep` + `Read`, paying no agent startup overhead to fetch one known thing. Safe because the initial sweep is for orientation and the downstream opus review loop + per-round challenge backstop any cheap-tier shallowness.

- **Rule written where each spawn-capable agent actually reads it** (subagents don't load SKILL.md): `review-protocol.md` gains a sixth "Reviewer shared discipline" (covers both reviewers when a spot-check fans out); `spec-implementer` gains a "pin the tier" key principle (it reads existing code to implement); `spec-author` gains the same note beside its economy paragraph (it may sweep the codebase to write design accurately). ## 1.8.0 (2026-07-03)

Restructure the execution architecture around a **generator/arbiter split** to cut top-tier-model token burn: the main agent (top-tier model) now only converses with the user, distills briefs, challenges subagent conclusions, escalates decisions, and maintains the review log; **all long-form generation moves to persistent subagent sessions** on a capable-but-cheaper tier. Motivation: with the main agent on a premium model and both reviewers on `model: inherit`, every design doc, plan, fix round, and review round burned premium tokens — the expensive model's actual value (judgment) was a small fraction of what it was being spent on (bulk writing).

- **New agent `spec-author` (opus)** — authors and revises planning/design documents per the main agent's brief. Mode 1: authoring (plan file in both modes; requirements/design/tasks + review-log skeleton in Spec Mode). Mode 2: issue-driven revision per a challenge-validated reviewer list. Replaces the main agent's direct Edit-tool authorship of design.md / the plan file everywhere. One author agent for both modes on purpose — planner and spec-doc-writer turned out to be the same job shape (write per brief / fix per issue list), differing only in target file.
- **Reviewers `model: inherit` → `opus`** — the single largest per-round saving. Depth is compensated by the new challenge exchange (below), not surrendered.
- **Persistent sessions (verified against official Claude Code docs: custom subagents are resumable via `SendMessage`; built-in Explore/Plan are not, which doesn't affect us)** — one reviewer session per loop and one author session per cycle, resumed each round instead of respawned: no re-reading review-protocol.md / steering / design every round, letter numbering accumulates naturally, and the Mode 2 fixer remembers why it wrote what it wrote. Mode A sparring continues into Mode B as the same session. Resume-failure fallback: fresh spawn rebuilt from review-log §1 (the log doubles as the loop's disaster-recovery checkpoint at zero extra cost).
- **Challenge exchange (rigor challenge)** — after every reviewer round's issue list (including `0 issues` rounds), the main agent sends exactly one substantive challenge in the reviewer session: dispute suspected false positives (demand the concrete scenario), probe suspected misses, question severity. The **final post-challenge list** is the round's official record (what enters review-log §1). Bounded to one exchange — a surviving disagreement is by definition a contested design choice and escalates through the existing Architecture Decision channel. This is the quality backstop that makes the reviewer downgrade safe: cheap generator, expensive critic.
- **Fidelity challenge** — after each `spec-author` Mode 1 delivery, the main agent checks the document against the brief for handoff drift (invented scope, dropped decisions, unflagged assumptions). Deliberately narrower than design review; it protects the conversation→document handoff before reviewer rounds are spent on the wrong document. The author must flag "Assumptions beyond the brief" explicitly in its report.
- **Convergence fuse extension** — on a Round-5 fuse trip, spawn one fresh reviewer instance for a single independent round before reporting: distinguishes "structural problem" (fresh eyes confirm) from "persistent-session anchoring / churn" (fresh eyes diverge). Keeps fresh-eyes as an escape hatch without paying its re-read cost every round.
- **Quick Fix Mode boundary redrawn** — plan authoring and plan fixes now go through `spec-author`; **code writing and code fixes stay with the main agent** (the mode's lightweight identity, kept deliberately — the code is small by definition and dispatch latency would erase the mode's point). Plan files now never live inside the repo: the harness path is handed to the author; when the environment provides none, the author creates `~/.claude/plans/<slug>.md` (previously `.spec/quickfix/<slug>.md`).
- **Main agent reading discipline** — read the artifact under review in full once, then per round read only the diffs. Challenging requires having read; it doesn't require re-reading the unchanged 90%.
- **Core Principles**: new shared #10 "Generation in Subagents, Arbitration in the Main Agent"; Spec/Quick-Fix principles renumbered 10→11…15→16; #15 renamed "Main Agent May Implement **Code**" to mark the narrowed exception.
- Docs updated in sync: SKILL.md (execution-architecture section, agents table, both mode flows, Quick Summary), review-protocol.md (new "Persistent sessions and the challenge exchange" section, main-agent responsibilities), both reviewer agents (first-round vs resumed-round workflow, challenge-response discipline), spec-workflow.md flow diagrams, mode-selection.md comparison table.
- **Quality pass (skill-creator review)**: fixed three leaks where document-edit ownership hadn't moved to `spec-author` (mode-selection's plan-file paragraph, post-briefing plan fixes in briefing-guide + SKILL.md, post-Decision design.md weaving in the Review Log Mechanism); surfaced the fresh-eyes fuse round in the documents the main agent actually reads (SKILL.md Quick Summary / Error Handling / both loop steps, plus review-protocol.md's own convergence summaries); taught `spec-author` the Quick Fix embedded `## Review Log` skeleton; cleaned pre-existing "AskUserQuestion stop point" briefing residue in spec-workflow.md diagrams (contradicted the 1.7.x turn-final two-beat discipline); added challenge/persistence bullets to the Quick Fix implementation-reviewer diagram; added `spec-author` to the README agents table.

## 1.7.6 (2026-07-01)

Fix the briefing checkpoint hook false-blocking the **compliant** flow — brief turn-final → user reply → ExitPlanMode — whenever a routine plan-mode marker happened to sit between the reply and the exit. A regression from the 1.7.1 anchor.

- **Root cause**: 1.7.1 anchored the check to "the current plan session" by treating a bare `{type:"permission-mode",permissionMode:"plan"}` marker as the (re-)entry boundary, assuming it only appears on a manual shift+tab re-entry. But Claude Code writes that **same** bare marker at essentially **every** plan-mode turn boundary (verified against live transcripts: it recurs while plan mode is already active, clustered with `last-prompt` / `mode` bookkeeping). So whenever such a routine marker fell between the user's reply and ExitPlanMode, the single backward walk hit the marker first and DENIED — false-blocking the exact flow the hook exists to allow. It bit reliably because the spec flow almost always does *something* after the user's approval (applies the tweak, updates the plan, loads the deferred ExitPlanMode) before exiting, re-emitting the marker.
- **Fix — split the single walk into two passes**. Pass 1 runs a small state machine to find where the current plan session started, using **reliable** signals only. A session *opens* on an `EnterPlanMode` tool call, a genuine `→plan` transition read from a **real user message's own `permissionMode` field** (a human action, so it can't be routine bookkeeping — the dependable signal for a manual re-entry, which carries no `EnterPlanMode` call), or a bare `permission-mode:plan` marker seen **while not already in plan**. A session *closes* on an **approved** ExitPlanMode (Claude Code delivers a "…approved your plan…" tool_result; a *denied* exit carries our own deny reason instead, so it does **not** close and recovery never deadlocks) or a real user message with a non-plan `permissionMode`. A bare `permission-mode:plan` marker seen **while already in plan is ignored** — that is the routine turn-boundary bookkeeping that caused the false-block. Pass 2 walks back from ExitPlanMode only to that boundary.
- **Kept intact on purpose**: a briefing **crammed into the same turn** as ExitPlanMode still DENIES — that message carries a tool_use so it is skipped as mechanical and the walk denies at the session boundary; a crammed mid-turn briefing may not even render, so blocking it is correct. Manual re-entry after a restart is still anchored via the reliable transition signal. Modeling session *close* (not just *open*) is what stops a later unbriefed shift+tab re-entry from leaking to the previous session's approval reply. **Residual** (accepted, all fail-open — err toward allow, never a false-block): the plan-entry scrolled off via compaction (no session identified → allow); a restart lands while already in plan with no intervening non-plan message (re-entry not detectable); and a shift+tab out-of-plan-then-back-in via bare markers with no typed message in between (bare non-plan markers are deliberately not treated as a session close, since doing so would let routine bookkeeping markers close a live session and re-introduce the very false-block this release removes). All three are consistent with the hook's fail-open discipline and strictly better than the false-block they replace.
- **Tests**: `briefing-checkpoint.test.js` grows 12 → 20. New cases pin: the reported false-block (routine marker between reply and exit → allow — **would have denied on the old hook**); manual re-entry detected via the user-message `permissionMode` transition (deny without a fresh briefing, allow with one); the cross-session leak (approved exit → unbriefed bare-marker re-entry → deny, vs briefed → allow); that a *denied* exit does not close the session so recovery still allows; and fail-open when no plan session is identifiable. `hooks.json` description updated in sync.

## 1.7.5 (2026-06-30)

Add a sixth review-residue pattern (Pattern F) extending the formal-doc isolation discipline to **production-code comments**, then close the cross-agent enumeration gaps the addition surfaced.

- **New Pattern F (`review-log-bad-examples.md`)**: production comments must not pin themselves to (A) review-log codes (`Decision X` / `Bug X` / `Smell X` / `Round-N` / `D<n>` / `Pivot-Event-N` / `SC-N`) riding inside otherwise-normal comments, nor (B) internal spec-doc pointers (`design.md §Component N` / a bare `Component N` / requirement IDs `R6.1` / `R13`). The unifying test is **durability**: a *stable identifier* may be cited; a *fragile pointer that drifts on doc reorganization* may not. **Allowed exceptions** (references that don't drift with the spec): an external standard (`RFC 6749 §5.2` / OAuth / IETF) and a spec's **name** (not its component numbers). Notably **`ADR-N` is *not* exempt**: in this project ADRs are `#### ADR-N:` sections inside design.md, so an ADR number drifts like `§Component 8` — citing it from code is type-(B) residue. Examples were pulled from real residue observed in a production project.
- **Enforcement wired so the discipline actually bites**: `implementation-reviewer`'s "review-residue in production code" Smell now catches both (A) and (B) — type-(B) had *zero* coverage before, so a docstring `see design.md §Component 13` previously sailed through; `spec-implementer`'s self-check carries the same ban list so the writer avoids emitting it in the first place. ## 1.6.6 (2026-06-15)

The ExitPlanMode briefing checkpoint changed from a **prompt hook to a deterministic command hook**, curing the 1.6.2–1.6.5 deadlock for good. (Note: this version number previously had a short-lived "remove the hook" commit that was pushed and then reverted via force-push, not retained on the remote; this 1.6.6 is the official release. If a stale 1.6.6 directory remains in the local plugin cache, it must be cleared and re-pulled to get this version's content.)

- **Root cause**: the prompt hook's judge is an LLM that can't see the conversation history. Even when instructed to "always allow", it still conservatively blocks ExitPlanMode because it "can't confirm whether the briefing was delivered" — blocking even when the briefing was correctly delivered and the user confirmed, causing a deadlock where the agent can't propose a plan (observed blocking twice in a row in practice). An LLM will override the prompt instruction itself, and can't serve as a reliable gate.
- **Solution**: switch to a Node command hook (`hooks/briefing-checkpoint.js`, deterministic code, not an LLM). Reads the transcript to judge "is the entry before ExitPlanMode a real user reply" — the turn-final briefing discipline guarantees a legitimate flow is always immediately preceded by a user reply; a skip is always assistant / tool_result. Yes → allow; skip → block and return a short reminder.
- **Scope narrowing (no hijacking of normal plan mode)**: although the hook is mounted on all ExitPlanMode, it only enforces **when "this plan cycle has run design-reviewer"** (= the spec-driven Quick Fix flow). The cycle start is delimited by **the most recent entry into plan mode** (`permission-mode:plan` / `mode:plan` / `EnterPlanMode` tool_use), and design-reviewer only counts if it comes after that start — calls between rounds (e.g. Spec Mode `/create-spec`, which also runs design-reviewer but not in plan mode) won't be miscounted. **If no cycle start is found, allow** (do not fall back to "the previous ExitPlanMode" as the boundary — that's the previous round's end and would fold the gap in, producing a false positive). Plain plan mode has no design-reviewer → always allow, so installing the plugin won't affect built-in plan mode.
- **No deadlock**: after the briefing + user reply, the preceding entry is a user message → allow; and **fail-open** — unreadable transcript / parse failure / not a Quick Fix cycle all allow, blocking only on "a confirmed skip inside a spec-driven cycle".
- **Detection details**: a subagent call appears in the transcript as an `Agent`/`Task` tool_use + `input.subagent_type`; the hook detects design-reviewer from this. It also filters out `isSidechain` (subagent-internal messages, otherwise false-deny) and `isMeta` (injected local-command / reminder, otherwise false-allow). A real human reply is `type:user` with string content.
- **Markers verified in practice** (against real transcripts): user manually shift+tabs into plan mode → `{type:"permission-mode", permissionMode:"plan"}`; agent enters via tool → `EnterPlanMode` tool_use. Both cycle-start paths are detectable, so "user manually enters plan mode first" won't cause a miss. (The `mode` field is always `"normal"` in a bypassPermissions session; the plan state lives in `permissionMode`.)
- **Cross-platform + stateless**: pure Node (shipped with Claude Code), works on Windows/Linux/Mac; only reads stdin + transcript (read-only), writes no file anywhere (including the project folder). The command uses `node "${CLAUDE_PLUGIN_ROOT}/hooks/briefing-checkpoint.js"`.
- briefing-guide's "three-layer reminder architecture" layer 3 is restated as a command hook; README Hooks section in sync.
